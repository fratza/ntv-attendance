# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

An attendance-tracking app (department selection → daily attendance grid) built with TanStack
Start, generated/managed via [Lovable](https://lovable.dev). The repo is synced with Lovable's
editor — see "Lovable sync" below before rewriting git history.

## Commands

Package manager is **bun** (see `bun.lock`, `bunfig.toml`).

- `bun run dev` — start the dev server (vite dev)
- `bun run build` — production build (vite build)
- `bun run build:dev` — build in development mode
- `bun run preview` — preview a production build
- `bun run lint` — eslint over the whole repo
- `bun run format` — prettier --write over the whole repo

There is no test suite/runner configured in this repo currently.

## Architecture

**Stack**: TanStack Start (file-based router + SSR) on Vite, React 19, Tailwind v4, shadcn/ui
("new-york" style, Radix primitives), TanStack React Query, Zod for search-param validation.

**Vite config (`vite.config.ts`)** uses `@lovable.dev/vite-tanstack-config`, which bundles the
TanStack Start plugin, React plugin, Tailwind, tsconfig-paths, and nitro (build target: cloudflare)
automatically. Do not add those plugins manually — duplicate plugin registration breaks the app.
Additional Vite config goes through `defineConfig({ vite: { ... } })`.

### Routing (`src/routes/`)

File-based routing via TanStack Router — see `src/routes/README.md` for the full convention table.
Key points:
- `src/routes/__root.tsx` is the only root layout/app shell; it renders `<Outlet />`, sets up
  `QueryClientProvider`, and defines the app's `notFoundComponent`/`errorComponent`.
- Every route file exports `Route = createFileRoute("/path")({...})`. Search params are validated
  with Zod (`validateSearch`), e.g. `src/routes/attendance.tsx` validates `?department=`.
- `src/routeTree.gen.ts` is auto-generated by the router plugin — never edit it by hand.
- Do not create `src/pages/`, `src/routes/_app/index.tsx`, or `app/layout.tsx` (Next.js/Remix
  conventions); they don't apply here.
- Current routes: `/` (`index.tsx`, department picker) → navigates to `/attendance?department=...`
  (`attendance.tsx`, per-day attendance grid with status codes P/L/A/WFH/AL/HL/SL/EL).

### SSR error handling

There's a deliberate multi-layer setup to make sure server-side errors never surface as a raw
`h3`/nitro JSON error, and are still reported to Lovable's error tracking:
- `src/start.ts` defines `errorMiddleware`, a `createMiddleware().server()` wrapper that catches
  request-handling errors and returns a rendered HTML error page instead of throwing raw.
- `src/lib/error-capture.ts` listens for global `error`/`unhandledrejection` events and stashes the
  last error (5s TTL) because h3 sometimes swallows an in-handler throw into a generic
  `{"unhandled":true,"message":"HTTPError"}` 500 response before `errorMiddleware` can catch the
  real error — `try/catch` alone won't see it.
- `src/server.ts` wraps the TanStack Start server entry's `fetch`: after calling it, it inspects
  500 responses for that swallowed-error JSON shape and, if found, replaces the response with a
  rendered error page while logging the real captured error.
- `src/lib/error-page.ts` renders the static HTML error page used by both of the above.
- Client-side React error boundaries call `reportLovableError` (`src/lib/lovable-error-reporting.ts`)
  which forwards to `window.__lovableEvents.captureException` if present (Lovable's runtime hook).

When touching server entry/error handling, preserve this chain — removing any one layer reopens a
path where raw h3 errors or unreported exceptions reach the user/response.

### UI components

`src/components/ui/` is shadcn/ui-generated (Radix + `class-variance-authority` + `tailwind-merge`).
Treat these as generated primitives — prefer composing them from route/feature code rather than
editing them, unless the actual primitive behavior needs to change. Path aliases (`@/*` →
`./src/*`, configured in `tsconfig.json` and `components.json`) are used throughout; use
`@/components`, `@/lib`, `@/hooks`, `@/components/ui` for new code rather than relative `../../`
paths.

## Lovable sync

This repo is connected to Lovable (see `AGENTS.md`). Avoid rewriting published git history — no
force-pushing, rebasing, amending, or squashing commits already pushed to the connected branch —
since that rewrites history on Lovable's side and can lose the user's project history. Keep the
connected branch in a working state, since every push syncs back into the Lovable editor.

## Dependency installs

`bunfig.toml` enforces a 24h "supply-chain guard": bun will skip installing package versions
published less than a day ago, except for the `@lovable.dev/*` packages already excluded there.
Don't add new entries to `minimumReleaseAgeExcludes` without confirming with the user first.
